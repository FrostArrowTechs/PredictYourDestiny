# Swiss Ephemeris 许可证评估

评估日期：2026-07-23

## 结论

当前项目以小型、盈利、闭源服务为目标，在签订 Swiss Ephemeris Professional License 前，不应集成、分发或部署 Swiss Ephemeris。免费许可采用 GNU AGPL；若不准备让整个组合程序满足 AGPL 的开放源码义务，应选择专业商业许可。最终许可边界应由 Astrodienst 书面确认，必要时交由法律顾问审核；本文不是法律意见。

因此，本阶段只完成许可与工程风险评估，不下载源码、绑定库或星历数据，也不开始生产集成。

## 官方依据

- 官方仓库说明 Swiss Ephemeris 采用 AGPL 或 Professional License 双重许可，并要求在分发软件或启用公共服务前作出选择。
- 官方许可页面和程序员手册同样列出 AGPL 与专业许可两条路径。
- 专业许可合同应作为商业使用范围、费用、更新和终止条款的最终依据，不能仅凭第三方封装库的许可证判断底层 Swiss Ephemeris 可否闭源使用。

## 工程与数据约束

- 在许可证确定前，不向 Go 服务加入 Swiss Ephemeris 绑定或间接依赖。
- 不默认下载完整 JPL 数据。官方仓库列出的 DE431/DE441 数据约为 2.6 GB，不适合在未确定精度需求和部署方案前纳入构建或运行环境。
- 获得许可后，只选择产品实际需要的行星、月球及时间跨度数据；固定数据清单、来源、版本、哈希和磁盘大小。
- 星历数据不得放入临时依赖缓存；部署镜像、持久数据目录和构建缓存必须分开管理。
- 高纬度宫制失败、历史时区缺失和星历范围外输入必须显式报错，不得静默切换简化算法。

## 集成准入条件

1. 明确选择 AGPL 路线，或取得覆盖本项目部署方式的专业许可书面文件。
2. 审计拟采用的语言绑定、动态库和数据文件各自的许可证与再分发条件。
3. 固定 Swiss Ephemeris、绑定库和星历数据版本。
4. 记录星历文件清单、哈希、来源、磁盘体积及更新流程。
5. 建立行星黄经、ASC/MC、宫位、相位及高纬度失败场景的黄金测试和误差阈值。
6. 验证开发、CI、预发布和生产环境能以同一版本可重复部署及回滚。

若专业许可成本或条款不符合产品阶段，应单独评估基于开放天文算法和 JPL 数据的替代实现；替代方案仍需重新核验数据许可、历史时区、计算精度与维护成本，不能视为自动合规。
